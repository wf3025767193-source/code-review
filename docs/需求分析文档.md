# AI 代码评审工具需求分析文档

## 1. 项目背景

在现代软件开发流程中，Pull Request 是团队协作、代码质量控制和知识共享的重要环节。开发者通常会通过代码评审发现潜在缺陷、改进代码设计、保证代码风格一致，并降低线上问题风险。

但随着项目规模扩大和迭代速度提升，传统人工代码评审逐渐暴露出以下问题：

- Pull Request 内容复杂，Reviewer 难以快速理解变更意图。
- 大型 PR 容易遗漏关键风险点。
- Review 质量依赖 Reviewer 经验，结果不稳定。
- 重复性检查占用大量时间，例如空值处理、异常处理、测试缺失等。
- 新成员对项目上下文不熟悉，参与 Review 成本较高。

因此，本项目计划开发一个 AI 辅助代码评审工具。用户指定 GitHub Pull Request 后，系统自动获取代码变更，结合上下文进行智能分析，生成 PR 变更总结、风险代码识别结果和 Review 建议，从而提升代码评审效率与质量。

## 2. 项目目标

本项目的目标是构建一个面向开发者的 AI 代码评审辅助工具，帮助 Reviewer 更快理解 PR 内容，更早发现潜在问题，并生成更具体、更可操作的 Review 建议。

核心目标包括：

- 自动获取 GitHub PR 的代码变更信息。
- 使用 AI 对 PR 进行结构化分析。
- 生成简洁准确的 PR 变更总结。
- 识别潜在风险代码并标注严重程度。
- 给出具体、可操作的 Review 建议。
- 降低误报和漏报，减少无价值评论。
- 提供清晰易用的结果展示页面。

## 3. 目标用户

### 3.1 普通开发者

普通开发者可以使用该工具在提交 PR 前进行自查，提前发现明显问题，减少来回修改成本。

### 3.2 Code Reviewer

Reviewer 可以通过工具快速了解 PR 的变更范围、重点风险和建议关注点，提升 Review 效率。

### 3.3 Tech Lead / Maintainer

Tech Lead 可以使用该工具辅助把控关键模块变更质量，尤其关注安全、性能、兼容性、架构影响等问题。

### 3.4 新成员

新加入项目的开发者可以借助 AI 生成的上下文总结，更快理解代码变更和项目逻辑。

## 4. 用户痛点分析

| 痛点 | 具体表现 | 工具解决方式 |
| --- | --- | --- |
| PR 理解成本高 | Reviewer 需要逐文件阅读 diff 才能理解整体意图 | 自动生成 PR 总结和影响范围 |
| 风险点容易遗漏 | 大 PR 或复杂逻辑中隐藏 bug 不易发现 | AI 识别风险代码并按严重程度排序 |
| Review 质量不稳定 | 不同 Reviewer 关注点不同 | 提供结构化检查维度 |
| 评论不够具体 | Review 评论可能只说“这里需要优化” | 生成包含原因和修改建议的评论 |
| 上下文不足 | 单独看 diff 难以理解业务逻辑 | 获取 PR 描述、文件上下文和相关代码片段 |
| Review 耗时较长 | 重复性检查浪费人工时间 | 自动完成初步检查和总结 |

## 5. 核心业务流程

```text
用户输入 GitHub PR URL
        ↓
系统解析仓库信息和 PR 编号
        ↓
调用 GitHub API 获取 PR 元数据和代码变更
        ↓
系统构建分析上下文
        ↓
AI 模型分析 PR 内容
        ↓
生成 PR 总结、风险点和 Review 建议
        ↓
前端展示结构化评审报告
```

## 6. 功能需求

### 6.1 PR 输入与解析

用户可以在系统中输入 GitHub Pull Request URL，系统需要自动解析以下信息：

- GitHub owner
- repository name
- pull request number

示例：

```text
https://github.com/example/project/pull/123
```

解析结果：

```json
{
  "owner": "example",
  "repo": "project",
  "pullNumber": 123
}
```

### 6.2 GitHub PR 信息获取

系统需要通过 GitHub API 获取 PR 相关信息，包括：

- PR 标题
- PR 描述
- PR 作者
- 源分支和目标分支
- commit 信息
- changed files
- diff / patch 内容
- 文件新增、删除、修改行数

对于私有仓库，系统应支持用户配置 GitHub Token。

### 6.3 PR 变更总结

系统应根据 PR 标题、描述、commit 信息和代码 diff 生成变更总结。

总结内容应包括：

- 本次 PR 的主要目的。
- 修改了哪些核心模块。
- 可能影响哪些功能。
- 是否包含测试、配置、依赖或数据库变更。

示例输出：

```text
本 PR 主要调整用户登录流程，新增验证码校验逻辑，并修改 AuthService 的异常处理方式。影响范围包括登录接口、认证服务和相关单元测试。
```

### 6.4 风险代码识别

系统应识别 PR 中可能存在风险的代码片段，并输出结构化结果。

风险类别包括：

- 功能缺陷
- 安全问题
- 性能问题
- 异常处理缺失
- 空值处理缺失
- 并发问题
- 数据一致性问题
- 兼容性问题
- 测试缺失
- 可维护性问题

每条风险结果应包含：

- 文件路径
- 行号或代码片段位置
- 严重程度
- 风险类型
- 问题描述
- 影响说明
- 修改建议
- 置信度

示例：

```json
{
  "file": "src/auth/login.ts",
  "line": 42,
  "severity": "high",
  "category": "security",
  "issue": "验证码校验失败后仍可能继续执行登录逻辑",
  "impact": "攻击者可能绕过验证码限制",
  "suggestion": "建议在验证码校验失败时立即 return 或抛出异常",
  "confidence": 0.86
}
```

### 6.5 Review 建议生成

系统应根据风险分析结果生成 Review 建议。

Review 建议需要满足：

- 指向具体文件和代码位置。
- 说明为什么存在问题。
- 给出可操作的修改建议。
- 区分必须修复和建议优化。
- 避免泛泛而谈的评论。
- 避免只针对代码风格提出低价值建议。

建议分类：

- `must_fix`：合并前必须修复。
- `should_fix`：建议修复，可能影响质量。
- `nice_to_have`：可选优化。

### 6.6 结果展示

系统应提供一个清晰的评审报告页面，展示内容包括：

- PR 基本信息
- 分析状态
- 分析耗时
- PR 变更总结
- 风险代码列表
- Review 建议列表
- 文件级分析结果
- 严重程度统计
- 模型置信度说明

页面应支持按以下条件筛选：

- 严重程度
- 风险类别
- 文件路径
- 是否必须修复

### 6.7 分析结果导出

MVP 阶段可以支持 Markdown 格式导出，方便开发者复制到 GitHub PR 评论区。

导出内容包括：

- PR 总结
- 高风险问题
- 建议修改项
- 分析说明

## 7. 非功能需求

### 7.1 准确性

系统应尽量减少误报和漏报。对于每条风险建议，需要输出置信度和依据，避免让用户难以判断建议价值。

### 7.2 响应速度

对于中小型 PR，系统应在可接受时间内完成分析。MVP 阶段目标：

- 小型 PR：10 秒内返回结果。
- 中型 PR：30 秒内返回结果。
- 大型 PR：支持分批分析或提示用户分析范围较大。

### 7.3 可用性

系统界面应简洁清晰，用户只需输入 PR URL 即可开始分析。分析结果应结构化展示，避免输出大段难以阅读的 AI 文本。

### 7.4 可扩展性

系统应预留扩展能力，未来可支持：

- GitLab / Gitee
- 自动评论到 GitHub PR
- CI/CD 集成
- IDE 插件
- 团队规则配置
- 多模型分析

### 7.5 安全性

系统应保护用户代码和访问凭证：

- GitHub Token 不应明文暴露在前端。
- 敏感配置应通过环境变量管理。
- 日志中不应记录完整 Token。
- 对私有仓库代码内容应避免长期保存。

## 8. AI 分析设计

### 8.1 模型选择思路

AI 代码评审任务对上下文理解和推理能力要求较高，因此模型选择应区分不同任务：

- PR 总结：使用响应速度快、成本较低的模型。
- 风险识别：使用代码理解和逻辑推理能力更强的模型。
- 报告整理：使用轻量模型生成结构化说明。

这种分层策略可以在准确性、成本和响应速度之间取得平衡。

### 8.2 上下文获取策略

为了避免模型只看 diff 导致理解不足，系统需要构建多层上下文：

- PR 标题和描述
- commit message
- changed files 列表
- 文件 diff
- 变更行附近代码上下文
- 相关测试文件
- 关键类型定义和被调用函数

MVP 阶段优先实现：

- PR 标题和描述
- changed files
- diff patch
- 变更文件路径
- 变更行附近上下文

未来可进一步加入：

- AST 分析
- 调用链分析
- 依赖图分析
- 历史缺陷记录
- 项目编码规范

### 8.3 Prompt 设计要求

Prompt 应要求模型输出结构化 JSON，便于系统解析和展示。

模型分析时应遵循以下原则：

- 只对 PR 变更相关问题提出建议。
- 必须引用具体文件和位置。
- 必须说明问题原因和影响。
- 不输出无法验证的结论。
- 对低置信度结果进行标记。
- 优先关注安全、数据、权限、并发、异常处理等高价值问题。

### 8.4 误报控制策略

为了降低误报，系统应采用以下策略：

- 设置置信度阈值，过滤低置信度建议。
- 要求模型给出具体代码依据。
- 对风格类建议默认降级。
- 对重复建议进行合并。
- 对无法定位到具体文件的建议不作为高风险问题展示。

### 8.5 漏报控制策略

为了降低漏报，系统应采用以下策略：

- 对高风险文件进行重点分析。
- 针对安全、权限、异常、数据一致性使用专项检查 Prompt。
- 对大型 PR 按文件分块分析，避免上下文过长导致遗漏。
- 对测试文件缺失进行单独检查。

## 9. MVP 范围

### 9.1 MVP 必做功能

MVP 阶段需要完成以下功能：

- 输入 GitHub PR URL。
- 自动解析 PR 地址。
- 调用 GitHub API 获取 PR 信息和 diff。
- 生成 PR 变更总结。
- 识别风险代码。
- 生成 Review 建议。
- 展示结构化分析报告。
- 支持 Markdown 导出。
- 提供模型选择和上下文策略说明。

### 9.2 MVP 可选功能

如果时间允许，可以增加：

- GitHub Token 配置。
- 私有仓库分析。
- 分析结果缓存。
- 按文件筛选风险。
- 一键复制 Review 评论。

### 9.3 MVP 暂不实现功能

以下功能可放入后续版本：

- 自动评论到 GitHub PR。
- 企业用户管理。
- 团队级规则配置。
- IDE 插件。
- 自训练模型。
- 全量代码仓库语义索引。
- 复杂调用链分析。

## 10. 系统边界

本系统定位为 AI 辅助代码评审工具，不替代人工 Reviewer 的最终判断。

系统应帮助用户：

- 更快理解 PR。
- 发现潜在风险。
- 提供 Review 参考。
- 减少重复性检查。

系统不保证：

- 发现所有 bug。
- 完全替代人工代码评审。
- 对业务逻辑做出百分百正确判断。
- 对缺少上下文的代码给出确定性结论。

## 11. 验收标准

MVP 完成后应满足以下验收标准：

- 用户可以输入合法 GitHub PR URL 并触发分析。
- 系统可以成功获取 PR 基础信息和 changed files。
- 系统可以生成不少于一段 PR 变更总结。
- 系统可以输出结构化风险列表。
- 每条风险建议包含文件、问题、影响、建议和严重程度。
- 页面可以清晰展示分析结果。
- 系统对无法访问的 PR 给出明确错误提示。
- 文档中说明模型选择、上下文获取和未来扩展设计。

## 12. 未来扩展方向

### 12.1 GitHub 深度集成

未来可以支持将 AI Review 建议自动发布为 GitHub PR Review Comment，并支持用户选择性提交评论。

### 12.2 CI/CD 集成

系统可作为 CI 流程的一部分，在 PR 创建或更新时自动运行分析，并输出检查结果。

### 12.3 团队规则配置

不同团队可以配置自己的 Review 规则，例如安全规范、代码风格、接口设计约束、数据库变更要求等。

### 12.4 代码知识库

系统可以构建项目级代码索引，理解模块依赖、调用链和历史修改记录，从而提升上下文理解能力。

### 12.5 多模型协作

未来可以采用多个模型分别负责总结、安全检查、性能检查、测试检查，并对结果进行聚合，提高分析质量。

### 12.6 反馈学习机制

用户可以对 AI 建议进行采纳、忽略或标记误报，系统根据反馈优化后续分析策略。

## 13. 总结

AI 代码评审工具的核心价值在于辅助开发者更快理解 PR、发现潜在风险并生成高质量 Review 建议。MVP 阶段应优先打通从 GitHub PR 获取代码变更到 AI 分析再到结构化报告展示的完整流程。

在设计上，系统需要重点关注分析准确性、上下文构建、误报与漏报控制、响应速度和使用体验。通过合理的模型选择、结构化输出和分层分析策略，该工具可以有效提升 Pull Request Review 的效率和质量。

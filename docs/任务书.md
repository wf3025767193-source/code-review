# AI 代码评审工具开发任务书

## 1. 任务目标

本任务书用于指导实现一个 AI 代码评审工具 MVP。系统需要支持用户输入 GitHub Pull Request URL，自动获取 PR 代码变更，并通过 AI 生成变更总结、风险代码识别结果和 Review 建议，最终在前端页面中展示结构化评审报告。

本项目定位为 AI 辅助 Review 工具，不替代人工 Reviewer。它的核心价值是帮助开发者快速理解 PR、定位高风险变更、生成更具体的 Review 建议。

## 2. 项目形态判断：单 Agent 还是多 Agent

### 2.1 结论

MVP 阶段建议采用“单 Agent + 多分析步骤”的方案；后续版本再演进为多 Agent 协作。

原因是 MVP 最重要的是跑通完整链路：获取 PR、构建上下文、调用模型、返回结构化报告。单 Agent 更容易控制成本、响应速度、调试复杂度和输出一致性。

### 2.2 单 Agent 方案

单 Agent 指系统中只有一个核心 AI 分析节点，由它完成 PR 总结、风险识别和 Review 建议生成。

适合 MVP 的原因：

- 架构简单，开发成本低。
- 模型调用次数少，响应速度更快。
- 输出格式更容易统一。
- 更容易定位错误和调试 Prompt。
- 对中小型 PR 已经足够可用。

单 Agent 的内部仍然可以拆成多个步骤：

```text
获取 PR 信息
    ↓
构建上下文
    ↓
整体总结
    ↓
风险识别
    ↓
建议生成
    ↓
结果校验与清洗
```

这些步骤可以在 LangGraph 中建模为一个顺序工作流，但不必拆成多个独立 Agent。

### 2.3 多 Agent 方案

多 Agent 指不同角色的 AI 节点分别负责不同审查任务，例如：

- Summary Agent：负责 PR 变更总结。
- Security Agent：负责安全风险检查。
- Logic Agent：负责业务逻辑和边界条件检查。
- Test Agent：负责测试缺失分析。
- Performance Agent：负责性能和资源使用分析。
- Aggregator Agent：负责合并、去重、排序和输出最终报告。

多 Agent 的优势：

- 每个 Agent 可以使用更明确的角色和 Prompt。
- 对复杂 PR 的覆盖面更广。
- 可以并行分析，提高大 PR 的处理效率。
- 便于后续扩展团队规则和专项检查。

多 Agent 的代价：

- 模型调用次数增加，成本更高。
- 结果可能重复或互相冲突，需要聚合器处理。
- 调试复杂度更高。
- 响应时间不稳定。
- 对 MVP 来说容易过度设计。

### 2.4 推荐演进路线

项目建议按以下路线演进：

```text
阶段一：单 Agent 顺序工作流
阶段二：单 Agent + 专项检查节点
阶段三：多 Agent 并行分析
阶段四：多 Agent + 项目知识库 + 团队规则
```

MVP 使用 LangGraph 的意义不是一开始就做复杂多 Agent，而是提前把流程建模成可扩展的图。这样后续把某个分析步骤替换成独立 Agent 会更自然。

## 3. 最终交付物

项目完成后应交付以下内容：

- 一个可运行的 Vue 3 前端页面。
- 一个基于 FastAPI 的 Python 后端服务。
- GitHub PR 信息获取能力。
- 基于 LangChain 的模型调用封装。
- 基于 LangGraph 的 Review 分析工作流。
- 结构化 Review 报告展示能力。
- Markdown 格式分析结果导出能力。
- 项目说明文档和运行说明。

## 4. 推荐技术方案

### 4.1 前端

前端使用：

- Vue 3
- Vite
- TypeScript
- Fetch API
- lucide-vue-next 图标库

前端主要负责：

- 输入 GitHub PR URL。
- 发起分析请求。
- 展示分析进度。
- 展示 PR 总结、风险列表和 Review 建议。
- 支持复制 Markdown 报告。

### 4.2 后端

后端使用：

- Python 3.11+
- FastAPI
- Uvicorn
- Pydantic
- httpx
- LangChain
- LangGraph

后端主要负责：

- 解析 PR URL。
- 调用 GitHub API 获取 PR 信息。
- 构造 AI 分析上下文。
- 使用 LangGraph 编排分析流程。
- 使用 LangChain 调用大模型。
- 对模型结果进行校验、清洗和降级。
- 向前端返回结构化 JSON。

### 4.3 AI 工作流框架分工

LangChain 负责：

- 封装模型调用。
- 管理 Prompt 模板。
- 处理结构化输出。
- 对接不同模型供应商。

LangGraph 负责：

- 编排分析流程。
- 管理工作流状态。
- 定义节点之间的流转关系。
- 后续扩展多 Agent 分析。

FastAPI 负责：

- 提供 HTTP API。
- 管理请求参数和响应模型。
- 处理异常。
- 对接前端。

## 8. LangGraph 工作流设计

### 8.1 MVP 工作流节点

MVP 推荐使用以下节点：

```text
start
  ↓
parse_pr_url
  ↓
fetch_pr_data
  ↓
build_context
  ↓
analyze_review
  ↓
validate_result
  ↓
finish
```

### 8.2 Graph State 设计

```python
from typing import TypedDict, Any

class ReviewState(TypedDict, total=False):
    pr_url: str
    owner: str
    repo: str
    pull_number: int
    pr_data: dict[str, Any]
    context: dict[str, Any]
    raw_analysis: str
    analysis: dict[str, Any]
    errors: list[str]
```

### 8.3 节点职责

`parse_pr_url`

- 解析 GitHub PR URL。
- 提取 owner、repo、pull number。

`fetch_pr_data`

- 调用 GitHub API。
- 获取 PR 标题、描述、作者、分支、changed files 和 patch。

`build_context`

- 过滤无关文件。
- 裁剪过大 patch。
- 构造模型输入上下文。

`analyze_review`

- 使用 LangChain 调用模型。
- 生成 PR 总结、风险列表和 Review 建议。

`validate_result`

- 校验模型输出是否符合 Pydantic Schema。
- 合并重复建议。
- 过滤低置信度风险。
- 计算风险统计指标。

## 9. Pydantic 数据模型

建议在 `backend/app/schemas/review.py` 中定义：

```python
from typing import Literal
from pydantic import BaseModel, Field

class RiskItem(BaseModel):
    file: str
    line: int | None = None
    severity: Literal["high", "medium", "low"]
    category: str
    issue: str
    impact: str
    suggestion: str
    confidence: float = Field(ge=0, le=1)

class ReviewSuggestion(BaseModel):
    file: str
    type: Literal["must_fix", "should_fix", "nice_to_have"]
    comment: str

class ReviewSummary(BaseModel):
    overview: str
    changedModules: list[str]
    impact: list[str]

class ReviewMetrics(BaseModel):
    highRiskCount: int
    mediumRiskCount: int
    lowRiskCount: int
    analyzedFileCount: int

class ReviewResult(BaseModel):
    summary: ReviewSummary
    risks: list[RiskItem]
    suggestions: list[ReviewSuggestion]
    metrics: ReviewMetrics
```

## 10. AI 分析策略

### 10.1 上下文构造

MVP 阶段模型输入应包含：

- PR 标题。
- PR 描述。
- 作者和分支信息。
- changed files 列表。
- 每个文件的 patch。
- 文件新增和删除行数。

对于过大的 PR，需要进行裁剪：

- 跳过 `package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`。
- 跳过 `dist`、`build`、`node_modules`。
- 跳过图片、二进制文件和构建产物。
- 单个 patch 超过限制时只截取关键部分，并在结果中说明。

### 10.2 Prompt 要求

Prompt 应要求模型：

- 只分析 PR 变更相关问题。
- 必须引用具体文件。
- 优先关注安全、异常处理、权限、数据一致性、并发和测试缺失。
- 不输出泛泛的代码风格建议。
- 每条风险必须包含严重程度和置信度。
- 输出必须符合 JSON Schema。

### 10.3 误报控制

实现时应注意：

- 置信度低于 0.5 的风险默认不展示为主要问题。
- 没有具体文件位置的问题不标为 high。
- 命名、格式、注释类建议默认归为 `nice_to_have`。
- 相同文件下相似建议应合并。
- 模型无法判断时应输出“不确定”，不要强行下结论。

### 10.4 漏报控制

实现时应注意：

- 对安全、权限、异常处理、数据一致性、测试缺失进行显式检查。
- 对删除测试、删除校验、删除鉴权逻辑的 diff 提高风险等级。
- 对核心目录如 `auth`、`payment`、`permission`、`database`、`security` 增加关注。

## 11. 开发阶段安排

### 阶段一：后端基础工程

任务：

- 初始化 FastAPI 项目。
- 配置 `.env` 读取。
- 实现 `/health`。
- 配置 CORS，允许前端访问。
- 编写 `requirements.txt`。

验收标准：

- `uvicorn app.main:app --reload --port 3001` 可以启动。
- 访问 `/health` 返回 `{ "status": "ok" }`。

### 阶段二：GitHub PR 获取模块

任务：

- 实现 `parse_pr_url.py`。
- 使用 `httpx` 调用 GitHub API。
- 获取 PR 基础信息和 changed files。
- 提取 patch 内容。
- 处理 Token 无效、PR 不存在、API 限流等错误。

验收标准：

- 输入合法 PR URL 后能获取 PR 标题、描述和文件变更。
- 输入非法 URL 时返回明确错误。

### 阶段三：LangChain 模型调用

任务：

- 封装模型客户端。
- 设计 Prompt 模板。
- 使用 Pydantic Schema 约束输出结构。
- 处理模型输出非 JSON 的情况。

验收标准：

- 后端可以根据 mock PR diff 返回结构化 ReviewResult。
- 模型输出异常时系统能返回可理解的错误。

### 阶段四：LangGraph 分析流程

任务：

- 定义 ReviewState。
- 实现工作流节点。
- 将 PR 获取、上下文构造、模型分析和结果校验串联。
- 在 `/api/review/analyze` 中调用 graph。

验收标准：

- 用户提交 PR URL 后可以得到完整 Review 报告。

### 阶段五：前后端联调

任务：

- 前端调用 `/api/review/analyze`。
- 展示 loading、错误和分析结果。
- 支持 Markdown 复制。

验收标准：

- 从输入 PR URL 到展示 Review 报告的完整链路可运行。

### 阶段六：完善与演示

任务：

- 完善 README 运行说明。
- 准备一个公开 PR 作为演示样例。
- 补充项目设计说明：模型选择、上下文策略、单 Agent 与多 Agent 演进路线。

验收标准：

## 14. 风险与应对

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| GitHub Token 无效 | 无法获取私有仓库 PR | 提供明确错误提示 |
| PR diff 过大 | 模型上下文超限或响应慢 | 文件过滤、分块分析 |
| 模型输出非 JSON | 前端无法渲染 | Pydantic 校验、重试或降级 |
| AI 建议误报 | 用户信任下降 | 输出置信度、降级低价值建议 |
| AI 漏报关键问题 | Review 质量不足 | 增加专项检查节点 |
| 多 Agent 结果冲突 | 报告不一致 | 使用 Aggregator 节点合并去重 |
| API 调用成本高 | 使用成本上升 | 缓存结果、区分模型任务 |

## 17. 实现建议

开发时建议先保证完整链路跑通，再逐步提升分析质量。第一版可以使用 mock AI 数据验证前端页面，随后接入真实 GitHub API 和模型 API。

后端不要一开始就做复杂多 Agent。推荐先用 LangGraph 搭建单 Agent 顺序工作流，把节点边界设计清楚。等 MVP 稳定后，再把风险识别拆成安全、测试、性能、业务逻辑等多个专项 Agent，并通过聚合节点统一去重、排序和输出。
